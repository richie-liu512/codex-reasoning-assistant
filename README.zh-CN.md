# Codex 智能推理助手

[English](README.md)

## 简介

Codex 智能推理助手是一个用于判断当前模型与推理等级是否适合任务的 Codex 插件。每次任务开始时，它会读取当前任务实际使用的模型和推理等级，分析接下来工作的难度与质量要求，并建议继续使用当前配置、提高或降低推理等级，或者更换模型。

对于多阶段任务，助手会在自然阶段边界重新判断下一阶段需要的配置。它优先保证任务质量，同时避免在明确、简单、容易验证的工作上继续使用没有实际收益的高推理等级。

## 主要功能

- 读取当前任务实际使用的模型和推理等级；
- 判断当前配置是否足以保证下一阶段的正确性、完整性、安全性和可验证性；
- 根据具体模型的能力与等级边界给出建议，而不是把不同模型上的同名等级视为相同能力；
- 当前等级不足时，在进入高风险阶段前提醒切换；当前等级明显偏高时，在合适的恢复点建议降低；
- 长线任务进入新阶段时重新评估，不让一次配置选择机械地贯穿整个任务；
- 无法确认实际等级时明确说明，不把配置默认值冒充为当前任务的实际值。

## 工作原理

插件由两个部分组成：

1. `UserPromptSubmit` Hook 在 Codex 开始处理任务前，只读获取当前模型和推理等级；
2. Skill 根据当前模型的能力定位、任务下一阶段的推理需求和验证难度，给出模型、推理等级与执行方式建议。

判断顺序始终是：

1. 先确认当前模型是否具备完成任务所需的能力；
2. 再选择能够保证结果质量的最低充分推理等级；
3. 最后决定直接继续、在困难阶段前暂停，还是在可恢复检查点建议降档。

如果模型本身不适合任务，单纯提高推理等级不能可靠弥补能力差距。

## 判断标准

助手判断的是**接下来的任务阶段**，而不是简单根据文件数量、任务时长或预计 Token 消耗决定等级。它会综合考虑目标和证据是否存在歧义、推理链是否深且脆弱、错误是否容易被测试发现、失败的影响与可逆性，以及工具、状态、并发和恢复过程是否复杂。

这些因素不是机械计分表。决定等级的核心问题是：这一阶段需要多少独立判断，以及判断错误后有多难发现和补救。

| 推理等级 | 通常适合的任务 |
| --- | --- |
| 无 / 极低 | 几乎不需要解释或判断的机械操作，并且当前模型支持这些等级。 |
| 低 | 路径明确、范围局部、影响较小，结果可以直接验证。 |
| 中 | 包含多个相互依赖的步骤和有限判断，但目标、验收标准与验证方法清楚。 |
| 高 | 需要追踪复杂逻辑、调试问题、检验关键假设或处理重要边界情况。 |
| 超高 | 目标或证据高度模糊、需要跨来源综合、失败影响较大，或者错误很难通过测试与复查发现。 |
| 最高 | 只用于最困难、质量优先的任务，并且代表性评测或既往失败已经证明它比超高等级更有收益。 |
| 极限 | 用于能够清晰拆分并适合主动委派给多个子代理的任务；它是编排模式，不只是更高的单线程推理等级。 |

同一个等级在不同模型上并不代表相同能力。当前策略对常见模型的处理原则是：

| 模型 | 建议起点与升级边界 |
| --- | --- |
| GPT-5.6 Sol | 普通非简单任务以中等级为平衡起点；复杂逻辑使用高等级，特别困难或高影响工作使用超高等级；最高等级需要实际评测证明收益。 |
| GPT-5.6 Terra | 日常多步骤任务通常从中等级开始，边界清楚的工作可用低等级；如果需要超高或最高等级，应先比较是否改用 Sol 更合适。 |
| GPT-5.6 Luna | 明确、重复和高容量工作通常使用低等级，有限判断可提高到中等级；如果开始需要高等级，应比较 Terra 或 Sol。 |
| GPT-5.4 mini / Codex Spark | 适合边界清楚、速度优先的任务；工作转为复杂调查、架构设计或高影响验证时，应优先更换模型，而不是持续堆高等级。 |
| 被既有流程固定的旧模型 | 保留已经验证有效的配置；允许迁移时，复杂阶段应比较较新的模型，不能假设提高等级可以消除模型能力差距。 |

在建议升档前，助手还会检查问题是否其实来自成功标准不清、上下文缺失、依赖关系未知、工具路径错误或缺少验证闭环。这些基础问题不能靠更高推理等级解决。

更详细的模型策略、复查机制和官方来源见 [推理策略](docs/reasoning-policy.md)。

## 安装

需要：

- 支持插件和生命周期 Hook 的 Codex；
- Python 3.10 或更高版本，无需安装第三方 Python 包。

添加插件市场：

```bash
codex plugin marketplace add richie-liu512/codex-reasoning-assistant --ref main
```

然后：

1. 打开 `/plugins`；
2. 选择 `codex-reasoning-assistant` 市场并安装 **Codex Reasoning Assistant（Codex 智能推理助手）**；
3. 打开 `/hooks`，检查并信任插件携带的 Hook；
4. 开启一个新任务。

如果你已经配置了另一个用于读取推理等级的 `UserPromptSubmit` Hook，请先停用其中一个，避免同一任务被重复检查。

## 使用方法

安装完成后，每次提交任务都会自动进行判断，不需要额外提示。典型结果如下：

```text
推理建议：当前 GPT-5.6 Sol · 高（实际值）｜下一阶段建议：Sol · 高｜继续执行
```

你也可以直接要求：

```text
检查当前模型和推理等级是否适合这个任务。
```

或者在长线任务的阶段边界询问：

```text
下一阶段建议使用什么模型和推理等级？
```

建议值与当前值相同，表示可以继续；建议更高，表示进入后续困难阶段前需要提高配置；建议更低，表示当前配置能够完成任务，但对剩余工作明显偏重。短任务通常会直接完成，长任务会在下一个可恢复检查点建议降低。是否切换始终由用户决定。

## 读取结果

- `actual（实际值）`：模型和推理等级已由当前任务状态确认；
- `default_only（仅默认值）`：只读取到 Codex 配置中的默认等级，尚未证明当前任务正在使用它；
- `unavailable（无法确认）`：当前环境无法可靠读取实际等级。

读取结果的可信度与配置是否适合任务是两个不同问题。插件会先说明读取结果是否可信，再独立判断当前配置是否合适。

## 数据与权限

检测过程只读取确认模型和推理等级所需的 Codex 元数据，不会输出、保存、记录或传输提示词正文，也不会写入 Codex 的任务状态。

详细说明见 [隐私说明](PRIVACY.md) 和 [安全说明](SECURITY.md)。

## 开发与验证

克隆仓库后可运行：

```bash
python plugins/codex-reasoning-assistant/scripts/verify.py
```

验证内容包括插件结构、Hook 包装器、隐私扫描、Python 语法和单元测试。

## 兼容性

Codex 的插件格式和内部状态结构可能继续变化。读取失败时，任务仍会继续，助手会把当前等级标记为无法确认，并只根据下一阶段任务给出建议。
